Understanding A2A Protocol: Agent-to-Agent Communication and Agent Discovery

Understanding A2A Protocol: Agent-to-Agent Communication

In the rapidly evolving landscape of AI systems, Agent2Agent (A2A) Protocol emerges as a critical standard for enabling seamless communication and collaboration between AI agents. This article explores what A2A is, how agent discovery works, and how it complements other protocols like the Model Context Protocol (MCP).

What is A2A?

The A2A protocol is an open standard developed by the Linux Foundation that enables standardized communication and collaboration between AI agents. Unlike traditional approaches where agents are wrapped as tools, A2A allows agents to interact directly as autonomous entities, regardless of their underlying framework or vendor.

The Problem A2A Solves

Without A2A, integrating these diverse agents presents significant challenges:

  1. Agent Exposure: Developers often wrap agents as tools, limiting their capabilities and preventing direct agent-to-agent negotiation
  2. Custom Integrations: Each interaction requires bespoke, point-to-point solutions
  3. Scalability Issues: Systems become difficult to scale and maintain as the number of agents grows
  4. Interoperability Gaps: This limits the organic formation of complex AI ecosystems
  5. Security Concerns: Ad hoc communication lacks consistent security measures

How A2A Solves These Problems

A2A addresses these challenges by establishing a standardized protocol for agents to:

  • Interact reliably and securely
  • Discover each other’s capabilities
  • Communicate without exposing internal implementation details
  • Support complex, multi-turn conversations and negotiations
  • Maintain autonomy while collaborating

Core Benefits of A2A

  1. Secure Collaboration: HTTPS-based communication with consistent security measures
  2. Interoperability: Breaks down silos between different AI agent ecosystems
  3. Agent Autonomy: Allows agents to retain their capabilities while collaborating
  4. Reduced Integration Complexity: Standardized communication reduces engineering overhead
  5. Support for Long-Running Operations: Native support for async execution and streaming

Key Design Principles

A2A follows enterprise-grade design principles:

  • Simplicity: Leverages existing standards like HTTP, JSON-RPC, and Server-Sent Events (SSE)
  • Enterprise Readiness: Robust authentication, authorization, security, privacy, tracing, and monitoring
  • Asynchronous: Native support for long-running tasks and streaming
  • Modality Independent: Agents can communicate using various content types
  • Opaque Execution: Agents collaborate without exposing internal logic, preserving IP and security

Agent Discovery: How Agents Find Each Other

For A2A-enabled agents to collaborate, they first need to discover each other and understand their capabilities. This is where the Agent Card comes in.

The Agent Card: A Digital Business Card for Agents

The Agent Card is a JSON document that serves as a standardized descriptor for an A2A Server (remote agent). It contains:

  • Identity: Name, description, and provider information
  • Service Endpoint: The URL for the A2A service
  • A2A Capabilities: Supported features like streaming or push notifications
  • Authentication: Required security schemes (e.g., Bearer, OAuth2)
  • Skills: Descriptions of available tasks with input/output modes and examples

1. Well-Known URI (Public Discovery)

Best for: Public agents or domain-specific discovery

Mechanism:

  • A2A Servers host their Agent Card at a standardized location: https://{agent-server-domain}/.well-known/agent-card.json
  • Client agents perform an HTTP GET request to this endpoint
  • The server returns the Agent Card as a JSON response

Advantages:

  • Easy to implement
  • Follows RFC 8615 standards
  • Facilitates automated discovery

2. Curated Registries (Catalog-Based Discovery)

Best for: Enterprise environments or public marketplaces

Mechanism:

  • A central registry service maintains a collection of Agent Cards
  • A2A Servers publish their cards to the registry
  • Client agents query the registry by skills, tags, or other criteria

Advantages:

  • Centralized management and governance
  • Capability-based discovery
  • Support for access controls and trust frameworks

3. Direct Configuration (Private Discovery)

Best for: Tightly coupled systems or development environments

Mechanism:

  • Client applications are directly configured with Agent Card information through config files or environment variables

Advantages:

  • Straightforward for known, static relationships

Disadvantages:

  • Inflexible for dynamic scenarios
  • Requires client reconfiguration for updates

Securing Agent Cards

Since Agent Cards may contain sensitive information, A2A recommends:

  • Authenticated Agent Cards: Use authenticated extended versions for sensitive data
  • Secure Endpoints: Implement access controls via mTLS, IP restrictions, or OAuth 2.0
  • Registry Selective Disclosure: Return different cards based on client identity and permissions
  • No Static Secrets: Use out-of-band dynamic credentials instead

Understanding the Difference

Model Context Protocol (MCP) is designed for:

  • Connecting AI models and agents to tools, APIs, and external resources
  • Standardizing how agents invoke stateless, well-defined functions
  • Typical use cases: LLM calling an external API, agent querying a database

Agent2Agent Protocol (A2A) is designed for:

  • Enabling independent agents to collaborate with each other
  • Supporting complex, multi-turn conversations and negotiations
  • Typical use cases: Customer service agent delegating to billing agent, travel agent coordinating with multiple service agents

The Key Distinction

  • MCP: Agents using capabilities (tools)
  • A2A: Agents partnering on tasks

Here’s a detailed comparison of the three approaches:

This comparison shows why A2A is particularly powerful for complex scenarios like intent-driven blockchain applications, where multiple autonomous agents need to negotiate and collaborate without pre-determined logic.

Why Both Protocols?

The distinction matters because:

  • Tools are typically primitives with static, well-defined functions
  • Agents are autonomous systems that reason, plan, maintain state, and engage in complex dialogue

Trying to represent an agent as a simple tool fundamentally limits its capabilities. A2A respects the agent’s autonomy while enabling rich collaboration.

  2. Authentication: Client requests a token from the auth server based on the Agent Card’s security schemes

Conclusion

The A2A protocol represents a significant step forward in AI agent collaboration, providing the standardization and security needed for complex multi-agent systems. By combining A2A for agent-to-agent communication with MCP for tool integration, developers can build more powerful, flexible, and interoperable AI systems.

The future of AI lies not in isolated intelligent systems, but in orchestrated ecosystems of autonomous agents working together to solve complex problems.

理解 A2A 协议：代理间通信和代理发现

在快速发展的 AI 系统领域，Agent2Agent (A2A) 协议作为一个关键标准出现，使 AI 代理之间能够无缝通信和协作。本文探讨了什么是 A2A、代理发现如何工作，以及它如何与模型上下文协议 (MCP) 等其他协议相互补充。

什么是 A2A 协议？

A2A 协议是由 Linux 基金会开发的开放标准，可以实现 AI 代理之间的标准化通信和协作。与传统方法（将代理包装为工具）不同，A2A 允许代理直接作为自主实体进行交互，无论其底层框架或供应商如何。

如果没有 A2A，集成这些多样化的代理会带来重大挑战：

  1. 代理暴露问题：开发人员经常将代理包装为工具，限制其能力并阻止直接的代理间协商
  2. 自定义集成：每次交互都需要定制的点对点解决方案
  3. 可扩展性问题：随着代理数量的增加，系统变得难以扩展和维护
  4. 互操作性缺陷：这限制了复杂 AI 生态系统的有机形成
  5. 安全顾虑：临时通信缺乏一致的安全措施

A2A 如何解决这些问题

A2A 通过建立一个标准化协议来解决这些挑战，使代理能够：

  • 可靠且安全地交互
  • 发现彼此的能力
  • 无需暴露内部实现细节就进行通信
  • 支持复杂的、多轮次的对话和协商
  • 在协作时保持自主性

A2A 的核心优势

  1. 安全协作：基于 HTTPS 的通信和一致的安全措施
  2. 互操作性：打破不同 AI 代理生态系统之间的壁垒
  3. 代理自主性：允许代理在协作时保持自身能力
  4. 降低集成复杂性：标准化通信减少工程开销
  5. 支持长时间运行的操作：原生支持异步执行和流式传输

关键设计原则

A2A 遵循企业级设计原则：

  • 简单性：利用现有标准如 HTTP、JSON-RPC 和服务器发送事件 (SSE)
  • 企业就绪：强大的身份验证、授权、安全性、隐私、追踪和监控
  • 异步：原生支持长时间运行的任务和流式传输
  • 模态独立：代理可以使用各种内容类型进行通信
  • 不透明执行：代理在不暴露内部逻辑的情况下协作，保护知识产权和安全性

Agent Card：代理的数字名片

Agent Card 是一个 JSON 文档，作为 A2A Server（远程代理）的标准描述符。它包含：

  • 身份信息：名称、描述和提供商信息
  • 服务端点：A2A 服务的 URL
  • A2A 能力：支持的功能，如流式传输或推送通知
  • 身份验证：必需的安全方案（如 Bearer、OAuth2）
  • 技能：可用任务的描述，包括输入/输出模式和示例

发现策略

A2A 定义了三种主要的发现策略：

1. 众所周知的 URI（公开发现）

适用于：公开代理或域特定的发现

机制：

  • A2A Server 在标准位置托管其 Agent Card：https://{agent-server-domain}/.well-known/agent-card.json
  • 客户端代理对此端点执行 HTTP GET 请求
  • 服务器以 JSON 响应返回 Agent Card

优势：

  • 易于实现
  • 遵循 RFC 8615 标准
  • 便于自动发现

2. 精选注册表（基于目录的发现）

适用于：企业环境或公共市场

机制：

  • 中央注册表服务维护 Agent Card 的集合
  • A2A Server 将其卡发布到注册表
  • 客户端代理按技能、标签或其他标准查询注册表

优势：

  • 集中管理和治理
  • 基于能力的发现
  • 支持访问控制和信任框架

3. 直接配置（私有发现）

适用于：紧密耦合的系统或开发环境

机制：

  • 客户端应用程序通过配置文件或环境变量直接配置 Agent Card 信息

优势：

  • 对于已知的静态关系，实现简单

劣势：

  • 动态场景中灵活性不足
  • 需要客户端重新配置以进行更新

保护 Agent Cards

由于 Agent Card 可能包含敏感信息，A2A 建议：

  • 经过验证的 Agent Card：对敏感数据使用经过验证的扩展版本
  • 安全端点：通过 mTLS、IP 限制或 OAuth 2.0 实施访问控制
  • 注册表选择性披露：根据客户端身份和权限返回不同的卡
  • 无静态密钥：使用带外动态凭证而非嵌入式静态密钥

理解两者的区别

Model Context Protocol (MCP) 设计用于：

  • 将 AI 模型和代理连接到工具、API 和外部资源
  • 标准化代理如何调用无状态的、定义明确的函数
  • 典型用例：LLM 调用外部 API、代理查询数据库

Agent2Agent Protocol (A2A) 设计用于：

  • 使独立代理能够相互协作
  • 支持复杂的、多轮次的对话和协商
  • 典型用例：客户服务代理委派给计费代理、旅游代理与多个服务代理协调

关键区别

  • MCP：代理使用能力（工具）
  • A2A：代理在任务上合作伙伴

综合对比：API vs MCP vs A2A

灵活性 低（代码写死） 中（标准接口，随插随用） 高（自主决策，动态博弈）
在区块链的应用 简单的 Oracle 喂价 AI 读取钱包私有账本 意图驱动的资产结算

这个对比展示了为什么 A2A 对于复杂场景（如意图驱动的区块链应用）特别强大，其中多个自主代理需要进行协商和合作，而无需预先确定的逻辑。

真实案例：汽车维修店

想象一家配备 AI 代理”技工”的自动汽车维修店：

  1. 客户交互 (A2A)：客户与”店铺经理”代理通信：”我的汽车有异响”

  2. 多轮诊断 (A2A)：经理询问后续问题：”能发送异响的视频吗？”或”漏液多久了？”

  3. 内部工具使用 (MCP)：机械师代理使用 MCP 与以下工具交互：

    • 车辆诊断扫描仪：scan_vehicle_for_error_codes(vehicle_id='XYZ123')
    • 维修手册数据库：get_repair_procedure(error_code, make, model)
    • 升降平台：raise_platform(height_meters=2)

  4. 供应商交互 (A2A)：机械师与零件供应商代理协商：”你们有丰田凯美瑞 2018 款的零件 #12345 吗？”

这个例子展示了 A2A 和 MCP 如何seamlessly 协作：

  • A2A 处理更高层级的、对话式的、面向任务的交互
  • MCP 让代理使用其特定的、结构化的工具

为什么需要两个协议？

这些区别很重要，因为：

  • 工具通常是具有静态的、定义明确的功能的原语
  • 代理是自主系统，能够推理、规划、维护状态并进行复杂对话

试图将代理表示为简单工具从根本上限制了其能力。A2A 尊重代理的自主性，同时支持丰富的协作。

A2A 请求生命周期

当客户端代理与远程 A2A 代理交互时，它遵循以下顺序：

sequenceDiagram
    participant Client as Client Agent
    participant AuthSvr as Auth Server
    participant A2ASvr as A2A Server<br/>(Remote Agent)
    
    rect rgb(200, 255, 200)
        Note over Client,A2ASvr: Agent Discovery
        Client->>A2ASvr: GET /.well-known/agent-card.json
        A2ASvr-->>Client: 返回 Agent Card<br/>(含安全方案信息)
    end
    
    rect rgb(200, 220, 255)
        Note over Client,AuthSvr: Authentication
        Client->>AuthSvr: 请求令牌<br/>(authorizationUrl)
        AuthSvr-->>Client: 返回 JWT Token
    end
    
    rect rgb(255, 230, 200)
        Note over Client,A2ASvr: Send Message (同步)
        Client->>A2ASvr: POST /sendMessage<br/>with JWT + Message
        A2ASvr-->>Client: 返回 Task Response<br/>(待处理)
    end
    
    rect rgb(255, 200, 200)
        Note over Client,A2ASvr: Send Message Stream (异步)
        Client->>A2ASvr: POST /sendMessageStream<br/>with JWT
        A2ASvr-->>Client: Task (Submitted)
        A2ASvr-->>Client: TaskStatusUpdateEvent (Working)
        A2ASvr-->>Client: TaskArtifactUpdateEvent (结果片段A)
        A2ASvr-->>Client: TaskArtifactUpdateEvent (结果片段B)
        A2ASvr-->>Client: TaskStatusUpdateEvent (Completed)
    end
  1. 代理发现：客户端从众所周知的 URI 或注册表获取 Agent Card
  2. 身份验证：客户端根据 Agent Card 的安全方案向身份验证服务器请求令牌
  3. sendMessage API：客户端发送同步请求并等待响应
  4. sendMessageStream API：对于长时间运行的操作，客户端通过服务器发送事件 (SSE) 流式传输结果

总结

A2A 协议代表了 AI 代理协作的重大进步,为复杂的多代理系统提供了所需的标准化和安全性。通过将 A2A 用于代理间通信，与 MCP 用于工具集成相结合，开发人员可以构建更强大、更灵活和更可互操作的 AI 系统。

AI 的未来不在于孤立的智能系统，而在于协调的自主代理生态系统一起工作来解决复杂问题。
